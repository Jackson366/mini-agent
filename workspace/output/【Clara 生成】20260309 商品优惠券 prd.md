### 名词解释 (如有)

| 术语/缩略词 | 说明 |
| ----------- | ---- |
| 满减券 | 满足一定金额条件后可减免指定金额的优惠券，如满100减20 |
| 折扣券 | 按比例折扣的优惠券，如9折券、8折券 |
| 新人券 | 仅限新注册用户领取和使用的优惠券 |
| 优惠券核销 | 用户在订单中使用优惠券并完成支付的过程 |
| 领取率 | 优惠券被用户领取的数量占发放总量的比例 |
| 使用率 | 已使用的优惠券数量占已领取优惠券数量的比例 |

## 四、项目背景与目标

### 背景描述

- 需求背景：为提升电商平台促销活动的转化率，需要设计一套简洁高效的商品优惠券功能，降低用户参与门槛，提高优惠券的领取和使用效率。
- 用户痛点：当前用户在参与促销活动时，优惠券领取流程复杂，操作步骤繁琐，导致用户流失和转化率下降。
- 目标用户：电商平台所有用户，包括新用户和老用户。

### 目标设定

- 核心目标：简化优惠券领取流程，提升优惠券领取率至 80%。
- 成功指标：
  - 优惠券领取率达到 80%
  - 优惠券领取流程步骤减少至 2 步以内
  - 用户领取时长控制在 5 秒以内

## 4.6 冲突与风险

### 已识别的冲突

- ⚠️ **与促销活动管理的功能重复**：优惠券应作为促销活动的一种形式，而非独立功能模块，避免数据孤岛和功能重复。
  - **解决方案**：在现有促销活动管理中扩展优惠券类型，统一管理促销活动。

- ⚠️ **角色权限配置的潜在冲突**：RoleInfo 可能主要用于系统操作权限，而非业务特权。
  - **解决方案**：使用 BuyerFeaturesEnum 管理业务特性，RoleInfo 仅用于系统权限。

### 技术风险

- ⚠️ **领取流程性能风险**：目标要求 5 秒内完成领取，需要优化接口性能。
  - **缓解措施**：优化数据库查询，使用缓存机制，异步处理非关键操作。

- ⚠️ **自动发放策略的复杂度**：自动发放需要设计灵活的规则引擎。
  - **缓解措施**：第一阶段先实现手动领取，第二阶段再扩展自动发放。

### 可行性说明

- **买家域**：高 - 现有数据模型支持扩展，实现复杂度简单
- **平台运营域**：高 - 可基于现有运营配置管理扩展，技术成熟

## 4.7 需求优化调整

基于领域专家建议，需求已做以下优化调整：

1. **优惠券定位调整**：优惠券作为促销活动的一种形式，而非独立功能，避免与现有促销活动管理重复。

2. **领取流程明确**：
   - 步骤 1：用户点击"领取优惠券"按钮
   - 步骤 2：系统自动将优惠券绑定到用户账号（无需额外操作）

3. **领取资格规则**：所有正常状态的用户均可领取，不区分用户类型。

4. **自动发放策略**：支持自动发放策略（如新用户注册自动发放、首单优惠等），提升运营效率。

5. **模板化能力**：提供常用优惠券模板（满减券、折扣券、新人券等），降低运营配置成本。

### 详细功能描述



#### 1. 一键领取优惠券

**功能概述**
用户在商品详情页或活动页点击"领取"按钮，系统自动将优惠券绑定到用户账号，整个流程在 5 秒内完成。

**输入**
- 用户 ID（已登录用户）
- 优惠券 ID

**输出**
- 领取成功提示
- 优惠券绑定到用户账号

**业务规则**
- 用户必须处于正常状态（非冻结、非黑名单）
- 优惠券必须处于进行中状态
- 优惠券库存充足
- 用户未超过领取次数限制
- 领取流程不超过 2 步：点击领取 → 自动绑定

**异常处理**
- 用户未登录：提示登录
- 用户状态异常：提示"当前账号状态异常，无法领取"
- 优惠券已领完：提示"优惠券已被领完"
- 超过领取次数：提示"已达到领取上限"

**界面要求**
- 领取按钮醒目，易于点击
- 领取成功后显示明确的成功提示
- 领取失败显示具体原因

#### 2. 优惠券列表查看

**功能概述**
用户可以查看已领取的优惠券列表，包括未使用、已使用、已过期等状态，并可按状态筛选。

**输入**
- 用户 ID
- 状态筛选条件（可选）

**输出**
- 优惠券列表，包括：
  - 优惠券名称
  - 面额/折扣
  - 使用条件
  - 有效期
  - 使用状态

**业务规则**
- 按领取时间倒序排列
- 未使用优惠券优先展示
- 已过期优惠券自动标记为过期状态

**异常处理**
- 无优惠券：显示空状态提示

**界面要求**
- 优惠券卡片清晰展示关键信息
- 状态标识明显（未使用/已使用/已过期）
- 支持状态筛选

#### 3. 优惠券配置管理

**功能概述**
运营人员可以基于模板快速创建优惠券，配置基础信息、发放规则和使用规则。

**输入**
- 优惠券模板选择（满减券/折扣券/新人券等）
- 基础信息：名称、类型、面额、有效期
- 发放规则：发放对象、发放数量、库存上限
- 使用规则：使用条件、适用商品、限领/限用数量

**输出**
- 优惠券配置保存成功
- 优惠券处于草稿状态

**业务规则**
- 优惠券只能在草稿状态下编辑
- 发布后不可修改核心配置（面额、类型）
- 有效期不能早于当前时间
- 发放数量不能超过库存上限

**异常处理**
- 配置项缺失：提示必填项
- 配置冲突：提示冲突原因
- 权限不足：提示无权限操作

**界面要求**
- 提供模板选择，降低配置成本
- 配置项分组展示，逻辑清晰
- 实时校验配置合法性

#### 4. 优惠券发放管理

**功能概述**
运营人员可以手动发放优惠券给指定用户，或配置自动发放策略。

**输入**
- 手动发放：优惠券 ID、用户列表
- 自动发放：优惠券 ID、发放规则（新用户注册/首单优惠等）

**输出**
- 发放成功数量
- 发放失败原因（如有）

**业务规则**
- 优惠券必须处于进行中状态
- 库存充足才能发放
- 自动发放策略触发时自动执行

**异常处理**
- 库存不足：提示库存不足
- 用户不存在：提示用户不存在
- 策略配置错误：提示配置错误

**界面要求**
- 手动发放支持批量导入用户
- 自动发放策略配置简洁明了
- 发放结果实时反馈

  - 是否为活跃用户（可领取更高额度优惠券）
  - 领取次数限制标识

**需要新增的功能：**
- **BuyerCouponInfo（买家优惠券信息）**：需要新增聚合根或实体，管理买家持有的优惠券：
  - 优惠券 ID
  - 领取时间
  - 使用状态（未使用/已使用/已过期）
  - 使用时间
  - 关联订单 ID

#### 三、冲突与问题识别

**发现的冲突：**
- ⚠️ **与现有角色权限管理的潜在冲突**：如果优惠券领取需要基于买家角色（如 VIP 买家、普通买家）进行差异化处理，需要确认 RoleInfo 是否支持优惠券相关的权限配置。当前 RoleInfo 可能主要用于系统操作权限，而非业务特权。

**发现的重复：**
- ⚠️ **买家绑定卖家关系的潜在重复**：如果优惠券是卖家发放的（如店铺优惠券），需要利用 BuyerBindSellerInfo 来判断买家是否有资格领取特定卖家的优惠券。需确认该功能是否已覆盖此场景。

**业务规则违反：**
- ⚠️ **买家状态管理的规则**：如果买家处于"冻结"或"黑名单"状态，不应允许领取优惠券。需确保优惠券领取逻辑与 BuyerStatusEnum 的状态判断一致。

**技术可行性：** 高
- 现有买家域数据模型已具备基础的买家信息管理能力，扩展优惠券相关功能不需要大规模重构。
- 优惠券领取逻辑可以作为买家域的一个新子模块，与现有功能解耦。

**数据模型影响：** 小调整
- 需要在 Buyer 聚合根下新增 BuyerCouponInfo 实体或值对象。
- 需要在 BuyerFeaturesEnum 中新增优惠券相关的特性枚举值。
- 不需要修改核心的 BuyerInfo、AccountInfo 等现有结构。

**实现复杂度：** 简单
- 优惠券领取流程主要涉及：
  1. 买家身份验证（已有）
  2. 领取资格判断（扩展 BuyerFeaturesEnum）
  3. 优惠券绑定（新增 BuyerCouponInfo）
- 不涉及复杂的业务逻辑或跨域协调。

#### 五、需求优化建议

**调整建议：**
1. **明确优惠券领取资格规则**：需求中提到"所有用户"，但实际业务中可能需要区分新用户、老用户、VIP 用户等。建议在 PRD 中明确不同用户群体的领取规则，避免后期频繁调整。
2. **简化领取流程的具体定义**：需求中提到"2 步以内"，建议明确这 2 步是什么：
   - 步骤 1：点击"领取优惠券"按钮
   - 步骤 2：系统自动绑定到买家账号（无需额外操作）
3. **优惠券使用场景的边界**：需求中未提及优惠券的使用规则（如适用商品范围、使用条件等）。建议在 PRD 中补充，确保买家域与订单域、商品域的协同逻辑清晰。

**替代方案：**
- **方案：自动发放优惠券**
  - 描述：买家登录或访问特定页面时，系统自动发放优惠券到买家账户，无需手动领取。
  - 优势：进一步简化流程，领取率可能接近 100%。
  - 劣势：可能导致优惠券滥用，增加成本；用户感知度较低，可能不重视优惠券。

**实施建议：**
- **优先级：** P0（核心功能，直接影响转化率目标）
- **实施顺序：**
  1. 第一阶段：实现基础的优惠券领取功能（买家点击领取 → 系统绑定）
  2. 第二阶段：扩展领取资格判断逻辑（新用户、老用户、VIP 等）
  3. 第三阶段：优化领取体验（如一键领取、批量领取等）

### 平台运营域专家分析

#### 一、平台运营域需求分析

**涉及的平台运营域功能：**
- 优惠券配置管理（创建、编辑、发布优惠券）
- 优惠券发放规则管理（发放时间、发放对象、发放数量）
- 优惠券使用规则管理（使用条件、适用商品、有效期）
- 优惠券数据统计与监控（领取率、使用率、核销情况）
- 运营活动配置（关联促销活动）

**核心目标：**
简化优惠券领取流程，提升优惠券领取率至 80%，降低用户操作门槛，提高促销活动转化率。

#### 二、现有功能映射

**可直接使用的现有功能：**
- **运营配置管理**：可用于配置优惠券的基础信息（名称、面额、有效期等）
- **权限管理**：可用于控制哪些运营人员可以创建和管理优惠券
- **平台运营事件管理**：可用于记录优惠券的创建、发布、修改等操作事件

**需要扩展的现有功能：**
- **运营配置管理**：需要扩展支持优惠券的发放规则配置（自动发放 vs 手动领取、发放对象筛选、库存管理）
- **平台运营状态管理**：需要扩展支持优惠券的状态管理（草稿、待发布、进行中、已结束、已失效）
- **运营范围管理**：需要扩展支持优惠券的适用范围配置（适用商品、适用品类、排除商品）

**需要新增的功能：**
- **优惠券模板管理**：需要新增优惠券模板功能，支持快速创建常用类型的优惠券
- **优惠券发放策略引擎**：需要新增自动发放策略配置（新用户注册自动发放、首单优惠、满额赠券等）
- **优惠券数据看板**：需要新增运营数据看板，实时监控优惠券的领取率、使用率、核销情况等关键指标
- **优惠券批量操作**：需要新增批量创建、批量发放、批量失效等批量操作功能

#### 三、冲突与问题识别

**发现的冲突：**
- ⚠️ **与现有运营配置管理的冲突**：如果优惠券配置过于复杂（如多层级的使用规则、复杂的适用范围），可能与"简化流程"的目标冲突。需要在功能丰富性和简洁性之间找到平衡。

**发现的重复：**
- ⚠️ **与现有促销活动管理的重复**：如果系统中已有促销活动管理功能，优惠券应作为促销活动的一种形式，而不是独立的功能模块，避免功能重复和数据孤岛。

**业务规则违反：**
- ⚠️ **权限控制缺失**：需求中未明确优惠券的创建、发布、修改权限控制规则。根据平台运营域的业务规则，不同角色的运营人员应有不同的操作权限（如初级运营只能创建草稿，高级运营可以发布）。

#### 四、可行性评估

**技术可行性：** 高
- 优惠券功能是电商平台的标准功能，技术实现成熟
- 可基于现有的运营配置管理和权限管理体系进行扩展
- 领取流程简化主要依赖前端交互优化和后端接口性能优化

**数据模型影响：** 小调整
- 需要在 PlatformOperator 聚合根下新增 CouponConfig（优惠券配置）子实体
- 需要扩展 PlatformOperatorEventEnum 增加优惠券相关事件（创建、发布、修改、失效等）
- 需要扩展 PlatformOperatorStatusEnum 增加优惠券状态（草稿、待发布、进行中、已结束、已失效）

**实现复杂度：** 中等
- 优惠券配置管理：简单（基于现有运营配置管理扩展）
- 优惠券发放策略引擎：中等（需要设计灵活的规则引擎）
- 优惠券数据看板：中等（需要实时数据统计和展示）
- 领取流程优化：简单（主要是前端交互优化）

#### 五、需求优化建议

**调整建议：**
1. **明确优惠券与促销活动的关系**：优惠券应作为促销活动的一种形式，而不是独立功能。建议在现有促销活动管理中扩展优惠券类型，避免功能重复。
2. **简化配置项，提供模板化能力**：为降低运营人员的配置成本，建议提供常用优惠券模板（如满减券、折扣券、新人券等），运营人员只需填写关键参数即可快速创建。
3. **增加权限分级管理**：建议明确不同角色的操作权限，如初级运营只能创建草稿，高级运营可以发布，超级管理员可以批量操作和失效。
4. **优化发放策略，支持自动化**：建议支持自动发放策略（如新用户注册自动发放、首单优惠、满额赠券等），减少人工干预，提升运营效率。

**替代方案：**
- **方案 A：轻量级优惠券**：只支持最基础的优惠券类型（满减券、折扣券），配置项极简，快速上线。优势是实现快、上线快；劣势是功能单一，后续扩展性差。
- **方案 B：标准优惠券**：支持常见的优惠券类型和配置项，提供模板化能力，支持自动发放策略。优势是功能完善、扩展性好；劣势是实现周期较长。

**实施建议：**
- **优先级**：P0（核心功能，直接影响促销活动转化率）
- **实施顺序**：
  1. 第一阶段：优惠券配置管理 + 手动领取流程（MVP）
  2. 第二阶段：优惠券模板 + 自动发放策略
  3. 第三阶段：数据看板 + 批量操作
