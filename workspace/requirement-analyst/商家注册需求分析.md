# 商家注册需求分析

**文档状态：** 草稿
**分析专家：** 需求分析专家
**日期：** 2026-03-02
**项目：** 用户中心项目 - 商家注册模块（MVP）

---

## 1. 需求摘要

### 1.1 业务目标
- 为B端商家提供快速、安全的注册通道，降低商家入驻门槛
- 收集商家核心信息（联系方式、店铺基本信息），为后续经营流程打基础
- 建立商家账户体系，支持后续登录、权限管理等核心能力

### 1.2 目标用户
- **主要用户：** 中小型电商商家、个体经营者
- **使用场景：** 首次入驻平台，需要创建账户并完成基本信息登记
- **核心诉求：** 注册流程简单快捷，信息收集不过度侵入

### 1.3 核心问题
- 如何在"简单注册"与"信息完整性"之间取得平衡？
- 商家注册后的初始状态应如何定义？（待审核/可直接使用/需补充资料）
- 是否需要支持多端（PC/移动端）注册？

## 2. 需求拆解

### 2.1 功能需求

#### FR1：注册入口与引导
- **描述：** 用户可通过平台入口进入注册页面
- **包含：**
  - 注册链接/按钮可见
  - 注册协议预知告知（用户协议、隐私政策）
  - 注册类型选择提示（个人商家/企业商家，若需区分）

#### FR2：手机号验证
- **描述：** 通过手机号+验证码完成身份验证
- **包含：**
  - 手机号输入与格式校验（11位数字）
  - 短信验证码发送与倒计时（60秒）
  - 验证码校验（6位数字，有效期5分钟）
  - 防刷机制（同手机号每日发送上限）

#### FR3：密码设置
- **描述：** 设置登录密码
- **包含：**
  - 密码强度校验（8-20位，包含字母+数字）
  - 密码明文/密文切换显示
  - 密码确认输入（二次验证）
  - 密码安全提示（弱密码警告）

#### FR4：店铺基本信息
- **描述：** 收集商家店铺核心信息
- **包含：**
  - 店铺名称（中文/英文，2-30字符）
  - 联系人姓名（2-20字符）
  - 联系人手机号（可与注册手机号不同）
  - 店铺类型（可选：个人/企业，若业务需要）

#### FR5：协议确认
- **描述：** 用户需勾选确认相关协议
- **包含：**
  - 用户协议勾选框（必选）
  - 隐私政策勾选框（必选）
  - 协议内容可点击查看（新窗口/弹层）
  - 未勾选时禁止提交

#### FR6：注册提交与反馈
- **描述：** 提交注册信息并返回结果
- **包含：**
  - 提交前全字段校验
  - 注册成功/失败反馈
  - 成功后跳转逻辑（登录页/完善资料页/首页）
  - 失败时错误提示（手机号已存在/验证码错误/系统错误）

### 2.2 非功能需求

#### NFR1：性能
- 注册接口响应时间 < 2秒
- 短信验证码发送延迟 < 5秒

#### NFR2：安全
- 密码加密存储（不可逆加密，如bcrypt）
- 短信验证码防爆破（每日发送上限、验证失败次数限制）
- 防SQL注入、XSS攻击

#### NFR3：可用性
- 支持主流浏览器（Chrome、Safari、Edge最新2个版本）
- 移动端适配（响应式布局）

#### NFR4：兼容性
- 手机号支持中国大陆（+86）号码段
- 后续需扩展支持港澳台及海外号码（P2）

### 2.3 约束条件

- **业务约束：**
  - 一个手机号只能注册一个商家账户
  - 店铺名称在平台内需唯一（是否需要？待确认）
- **技术约束：**
  - 需对接短信服务供应商（如阿里云、腾讯云）
  - 需对接用户中心数据库/服务
- **法规约束：**
  - 需符合《网络安全法》《个人信息保护法》要求
  - 需明示个人信息收集范围与用途

## 4. 功能清单（MVP）

### 4.1 P0（必须有）
- [x] 手机号输入与格式校验
- [x] 短信验证码发送与校验
- [x] 密码设置与确认
- [x] 店铺名称输入
- [x] 联系人信息输入
- [x] 用户协议与隐私政策勾选
- [x] 注册提交与成功/失败反馈
- [x] 注册成功后自动登录/跳转登录页

### 4.2 P1（应该有）
- [ ] 密码强度提示与安全建议
- [ ] 密码明文/密文切换显示
- [ ] 短信验证码倒计时显示
- [ ] 手机号已存在/已被注册的提示
- [ ] 注册失败时的详细错误信息（字段级/接口级）
- [ ] 用户协议与隐私政策查看弹层

- [ ] 注册后引导完善资料（如店铺logo、简介等）
- [ ] 支持多手机号区号（港澳台/海外）

## 5. 验收标准

### 5.1 功能验收标准

#### AC1：手机号验证
**Given** 用户在注册页面
**When** 输入非11位数字的手机号
**Then** 显示"手机号格式不正确"的错误提示

**Given** 用户在注册页面
**When** 输入正确的11位手机号并点击"获取验证码"
**Then** 系统发送6位验证码短信，按钮进入60秒倒计时

**Given** 用户收到验证码短信
**When** 输入正确的6位验证码
**Then** 验证通过，可继续下一步

**Given** 用户在注册页面
**When** 输入错误的验证码
**Then** 显示"验证码错误"的提示，可重新输入

#### AC2：密码设置
**Given** 用户在注册页面
**When** 输入少于8位的密码
**Then** 显示"密码长度需为8-20位"的错误提示

**Given** 用户在注册页面
**When** 输入仅包含数字的密码
**Then** 显示"密码需包含字母和数字"的错误提示

**Given** 用户输入了密码
**When** 确认密码与密码不一致
**Then** 显示"两次输入的密码不一致"的提示

#### AC3：店铺信息
**Given** 用户在注册页面
**When** 店铺名称为空
**Then** 显示"请输入店铺名称"的错误提示

**Given** 用户在注册页面
**When** 店铺名称超过30个字符
**Then** 显示"店铺名称不能超过30个字符"的提示

**Given** 用户在注册页面
**When** 联系人姓名为空
**Then** 显示"请输入联系人姓名"的提示

#### AC4：协议确认
**Given** 用户在注册页面
**When** 未勾选用户协议或隐私政策
**Then** "注册"按钮不可点击或提交时显示"请先阅读并同意协议"的提示

#### AC5：注册提交
**Given** 用户填写完所有必填信息
**When** 点击"注册"按钮
**Then** 系统校验所有字段，校验通过后提交注册请求

**Given** 注册成功
**When** 服务器返回成功响应
**Then** 显示"注册成功"提示，3秒后自动跳转至登录页或自动登录

**Given** 注册失败
**When** 服务器返回错误响应（如手机号已存在）
**Then** 显示对应的错误提示，用户可修改后重新提交

### 5.2 非功能验收标准

| 类型 | 验收标准 | 验证方法 |
|------|----------|----------|
| 性能 | 注册接口响应时间 < 2秒 | 压力测试 |
| 性能 | 短信验证码发送延迟 < 5秒 | 功能测试 |
| 安全 | 密码加密存储 | 代码审查、数据库检查 |
| 安全 | 短信验证码防爆破 | 渗透测试 |
| 可用性 | 支持Chrome、Safari、Edge最新2个版本 | 浏览器兼容测试 |
| 兼容性 | 移动端响应式布局正常显示 | 设备测试（iOS/Android） |

## 6. 优先级建议

### 6.1 必做（P0）- MVP核心能力
**理由：** 这些功能构成商家注册的最小闭环，缺少任何一个都无法完成基本的注册流程。

- 手机号+验证码验证（身份验证基础）
- 密码设置（账户安全基础）
- 店铺基本信息（业务信息最小集合）
- 协议确认（法规合规基础）
- 注册提交与反馈（用户体验完整性）

### 6.2 应做（P1）- 提升体验与安全性
**理由：** 这些功能能够显著提升用户体验和系统安全性，但短期内可以牺牲。

- 密码强度提示（提升账户安全）
- 短信验证码倒计时（防止重复点击，降低成本）
- 详细错误提示（降低用户困惑）
- 协议内容查看弹层（合规透明）

### 6.3 可做（P2）- 增值功能
**理由：** 这些功能能够提供更好的用户体验或运营能力，但可以在后续迭代中实现。

- 注册后引导完善资料（提高资料完整度）
- 多手机号区号支持（扩展海外商家）
- 注册来源追踪（运营分析支持）
- 第三方快捷登录（降低注册门槛）

### 6.4 暂缓（P3）- 非MVP范围
**理由：** 这些功能属于企业级商家或高级运营需求，与"简单注册"的定位不符。

- 企业认证（营业执照核验）
- 批量导入商家
- 邀请码机制

## 7. 风险与待确认

### 7.1 需求澄清风险

| 风险项 | 影响 | 待确认事项 | 建议 |
|--------|------|------------|------|
| 注册后的账户状态 | 高 | 注册后是否需要审核？能否直接使用？ | 建议MVP阶段注册成功即可使用，后续增加审核机制 |
| 店铺名称唯一性 | 中 | 是否要求店铺名称在平台内唯一？ | 建议MVP不强制唯一，后续可增加重复提示 |
| 多端注册支持 | 中 | 是否需要同时支持PC和移动端？ | 建议MVP先做响应式，支持两端 |
| 联系人手机号 | 低 | 联系人手机号是否必须与注册手机号一致？ | 建议允许不同，增加灵活性 |

### 7.2 技术实现风险

| 风险项 | 影响 | 缓解措施 |
|--------|------|----------|
| 短信服务稳定性 | 高 | 选择可靠的短信供应商（如阿里云、腾讯云），设置备用通道 |
| 验证码防刷 | 中 | 实施发送频率限制、IP限制、图形验证码 |
| 密码加密强度 | 高 | 使用行业标准加密算法（如bcrypt），定期安全审计 |
| 手机号已注册判断 | 中 | 提供友好的"手机号已注册，是否直接登录？"提示 |

### 7.3 业务流程风险

| 风险项 | 影响 | 缓解措施 |
|--------|------|----------|
| 注册后流失 | 中 | 注册后提供引导流程，明确下一步操作 |
| 信息不完整 | 中 | 注册后引导完善资料，设置资料完整度提醒 |
| 恶意注册 | 低 | 增加设备指纹、行为分析等风控手段（P2） |

## 8. 给 Main 的交付建议

### 8.1 与其他子智能体的协作建议

1. **卖家域专家**：建议咨询卖家域专家，确认商家注册后的典型流程（如是否需要立即完善商品、是否需要设置支付方式等），以便定义注册后的跳转逻辑。

2. **平台运营域专家**：建议咨询平台运营域专家，确认是否需要在注册流程中加入"注册来源追踪""邀请码"等运营相关功能。

3. **UI/UX 专家**：建议咨询UI/UX专家，确认注册页面的信息层级、交互流程和错误提示方式，以提升用户体验。

### 8.2 需求边界建议

建议将本次需求边界限定为：
- **包含：** 手机号验证、密码设置、店铺基本信息、协议确认、注册提交
- **不包含：** 企业认证、店铺完善资料、商品发布、支付配置等后续流程

### 8.3 验收标准建议

建议采用"逐步验收"方式：
1. 第一阶段：完成P0功能的基本验收（功能可用、无明显Bug）
2. 第二阶段：完成P1功能的验收（体验优化、安全性增强）
3. 第三阶段：完成非功能性验收（性能、安全、兼容性测试）

### 8.4 待 Main 决策的事项

以下事项需要 Main 进行最终决策：

1. **注册后账户状态：** 注册成功后是否需要审核？能否直接使用？
2. **店铺名称唯一性：** 是否要求店铺名称在平台内唯一？
3. **联系人手机号规则：** 联系人手机号是否必须与注册手机号一致？
4. **MVP范围确认：** 是否同意将P2功能移至后续迭代？
5. **多端支持策略：** 是否采用响应式设计同时支持PC和移动端？
