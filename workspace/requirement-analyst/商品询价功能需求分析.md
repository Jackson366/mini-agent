## 一、业务背景与目标

### 1.1 业务背景

当前电商平台买家在批量采购场景下存在以下痛点:

- **逐个询价效率低**: B端采购商需要联系多个商家,沟通成本高
- **议价链路不透明**: 缺乏标准化的议价协商流程
- **价格有效期管理缺失**: 商家报价后缺乏有效期约束,导致后续履约争议
- **历史议价记录不可追溯**: 买家无法查看历史议价记录,难以做价格对比

### 1.2 业务目标

| 目标维度 | 具体指标 |
|---------|---------|
| **转化率提升** | 批量询价场景下,采购订单转化率提升 20% |
| **用户体验** | 批量询价操作路径缩短至 3 步以内 |
| **商家响应率** | 商家 24 小时内响应率达到 85% |
| **GMV 增长** | B 端采购商 GMV 季度环比增长 15% |

### 1.3 核心价值主张

- **买家侧**: 一次发起批量询价,统一管理议价进程,透明可追溯
- **商家侧**: 集中处理询价请求,灵活报价策略,提升批量成交率
- **平台侧**: 完善批量采购场景,增强 B 端买家粘性,提升平台 GMV

## 二、用户角色定义

### 2.1 角色矩阵

| 角色类型 | 核心特征 | 核心诉求 | 使用频率 |
|---------|---------|---------|---------|
| **C 端个人买家** | 单次采购量小,价格敏感度中等 | 快速比价,简单议价 | 低频(季度 1-2 次) |
| **B 端采购商** | 批量采购,议价能力强,决策周期长 | 批量询价,多轮议价,价格稳定 | 高频(月度 5-10 次) |
| **商家(卖家)** | 需平衡利润与成交量,响应能力有限 | 灵活报价,批量优惠,议价效率 | 高频(日均 10-30 次) |

### 2.2 角色详细画像

#### 2.2.1 C 端个人买家

- **典型场景**: 家装采购、活动物资采购、礼品批量采购
- **核心诉求**:
  - 快速找到价格合适的商家
  - 了解市场平均价格水平
  - 获得少量批量采购优惠
- **操作特点**:
  - 偏好移动端操作
  - 期望流程简洁,无需复杂配置
  - 对响应时效要求较高

#### 2.2.2 B 端采购商

- **典型场景**: 企业办公采购、工程项目采购、库存补货
- **核心诉求**:
  - 批量询价,统一议价管理
  - 多商家报价对比
  - 建立长期合作关系,获得协议价
  - 历史议价记录可追溯
- **操作特点**:
  - 偏好 PC 端精细化管理
  - 需要导出询价/报价单据
  - 对议价规则和流程有明确要求

#### 2.2.3 商家(卖家)

- **核心诉求**:
  - 快速识别高价值询价请求
  - 灵活设置批量折扣规则
  - 主动管理议价有效期
  - 批量处理同类询价
- **操作特点**:
  - 需要移动端快速响应 + PC 端深度管理
  - 期望设置自动报价规则,减少人工干预
  - 对成交转化率敏感

## 五、非功能需求

### 5.1 性能要求

| 指标 | 要求 | 说明 |
|------|------|------|
| **询价单创建** | ≤ 2 秒 | 从提交到显示成功状态 |
| **报价提交** | ≤ 2 秒 | 商家提交报价响应时间 |
| **消息推送延迟** | ≤ 5 秒 | 状态变更到消息推送到达 |
| **列表加载** | ≤ 1.5 秒 | 询价单/报价列表首次加载 |
| **并发支持** | 1000 QPS | 峰值询价并发处理能力 |

### 5.2 可用性要求

| 指标 | 要求 |
|------|------|
| **系统可用性** | ≥ 99.9% |
| **数据一致性** | 议价价格与订单价格强一致 |
| **消息到达率** | ≥ 99% |

### 5.3 安全要求

| 风险点 | 防护措施 |
|-------|---------|
| **恶意询价** | 同一买家单日询价次数上限(50 次) |
| **价格篡改** | 议价价格加密传输,下单时二次校验 |
| **越权操作** | 询价单/报价单访问权限校验 |
| **数据泄露** | 议价信息仅买卖双方可见 |

### 5.4 通知时效要求

| 场景 | 时效要求 | 通知渠道 |
|------|---------|---------|
| **商家接收询价** | 实时(≤ 5 秒) | 站内消息 |
| **买家接收报价** | 实时(≤ 5 秒) | 站内消息 + App 推送 |
| **议价即将过期** | 提前 2 小时 | 站内消息 |
| **商家响应超时** | 24 小时未响应时 | 站内消息 + 短信(可选) |

## 六、验收标准

### 6.1 功能验收

#### 6.1.1 批量询价流程

- [ ] 买家可从购物车/详情页批量选择 1-20 个商品发起询价
- [ ] 询价单包含:商品清单、期望价格、采购数量、备注
- [ ] 提交成功后显示询价单详情页,状态为"待商家响应"

#### 6.1.2 商家报价流程

- [ ] 商家在消息中心收到询价通知
- [ ] 点击进入询价详情,可查看所有商品和买家期望价格
- [ ] 支持按商品设置报价价格和有效期(1-30 天)
- [ ] 支持拒绝询价并选择/填写原因

#### 6.1.3 多轮议价流程

- [ ] 买家收到报价后,可选择"接受/继续议价/拒绝"
- [ ] 继续议价需输入新期望价格,系统记录议价轮次
- [ ] 议价达到 5 轮后,提示"议价次数已达上限,请接受报价或取消"
- [ ] 议价历史按时间线展示,包含每次报价/议价的价格和时间

#### 6.1.4 下单转化流程

- [ ] 买家接受报价后,跳转至订单确认页
- [ ] 订单价格自动填充为议价成交价
- [ ] 价格在有效期内锁定,过期后恢复原价并提示

### 6.2 边界条件验收

#### 6.2.1 商家不响应场景

- [ ] 询价提交 24 小时后,若商家未响应,状态变更为"已超时"
- [ ] 买家收到超时通知,可选择"重新发起询价"或"取消"

#### 6.2.2 议价失败场景

- [ ] 商家拒绝议价后,状态变更为"已拒绝",买家可查看拒绝原因
- [ ] 买家主动取消议价后,状态变更为"已取消"

#### 6.2.3 价格有效期场景

- [ ] 报价有效期截止前 2 小时,买家收到即将过期提醒
- [ ] 有效期过期后,议价价格失效,恢复商品原价
- [ ] 买家在有效期内下单,使用议价价格;过期后下单提示"议价已过期"

#### 6.2.4 库存变动场景

- [ ] 询价期间若商品库存不足,买家收到"库存不足"提示
- [ ] 商家报价时可设置"库存不足"为拒绝原因

### 6.3 数据统计验收

- [ ] 询价单数据包含:发起时间、买家 ID、商家 ID、商品清单、状态、议价轮次、成交价格
- [ ] 商家可查看:询价量、响应率、成交率、平均议价轮次
- [ ] 平台可查看:整体询价转化漏斗、平均响应时间

## 七、风险与建议

### 7.1 风险识别

| 风险类型 | 风险描述 | 应对建议 |
|---------|---------|---------|
| **商家响应率低** | 商家议价意愿不足,导致买家体验差 | • 引入商家评分机制<br>• 对响应率高的商家给予流量倾斜<br>• 设置最低响应率考核 |
| **议价成本高** | 大量低价值询价浪费商家时间 | • 设置询价门槛(如单笔金额 ≥ 100 元)<br>• 引入买家信用分,限制恶意询价 |
| **价格纠纷** | 议价价格与实际成交价不一致 | • 议价价格加密存储<br>• 下单时二次校验<br>• 保留议价记录作为证据 |
| **性能瓶颈** | 批量询价高峰期系统压力大 | • 引入消息队列削峰<br>• 优先保证核心链路可用<br>• 设置降级开关 |

### 7.2 需求澄清建议

以下问题需要与产品/业务进一步确认:

1. **跨店铺询价**: 是否允许买家向不同店铺的商品发起统一询价?若允许,商家如何响应?
2. **议价次数限制**: 5 轮议价上限是否合理?是否需要支持商家主动调整?
3. **价格到期处理**: 议价价格到期后,订单是否自动取消?还是需要买家重新确认?
4. **自动报价规则**: 是否默认对所有商家开放?是否需要审核机制?
5. **B 端专属权益**: 是否对 B 端采购商提供更低折扣、更长有效期等差异化权益?

### 7.3 后续迭代方向

- **智能议价助手**: 基于历史数据为买家提供议价价格建议
- **批量议价**: 商家批量调整多个买家的议价请求
- **议价模板**: 常用议价话术模板,提升沟通效率
- **视频议价**: 支持买卖双方视频沟通议价(高价值商品场景)

### 8.2 状态机定义
