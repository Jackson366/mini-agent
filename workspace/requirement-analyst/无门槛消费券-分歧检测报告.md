## 一、详细冲突分析（按严重程度排序）

### 🔴 冲突1: 退款时券是否返还

#### 涉及专家
- **buyer-domain**: 支持返还券（未发货时）
- **seller-domain**: 强烈反对返还券
- **platform-operations**: 倾向不返还

#### 各方观点

**buyer-domain 观点**（第289-303行）:
```
推荐方案:
✅ 退款金额 = 实付金额 34元
✅ 券的状态 = 退回到用户账户，恢复原有效期
✅ 提示文案："退款成功，5元券已返还，有效期至X月X日"

关键逻辑:
- 券返还后，有效期从原到期时间计算（不延长）
- 如果券已过期，则不返还，提示"券已过期，无法返还"
```
**理由**: 用户最关心退款场景，返还券能降低用户流失率，提升用户满意度

**seller-domain 观点**（第112-137行）:
```
场景2: 已发货/已收货后退款
退款金额: 90元(返还用户)
券处理: 不返还,券已消耗
平台承担损失: 7元
商家承担损失: 3元

商家实际结算:
= 用户实付90元
- 商家承担券成本3元
- 平台佣金4.5元
- 退款返还用户90元
= 商家净亏损: -7.5元 (含佣金损失和券成本)
```
**理由**: 
- 返还券会导致商家承担双重损失（券成本 + 退货损失）
- 用户可能恶意"试单"后退款，导致商家退货率上升
- 严重性: ⭐⭐⭐⭐⭐

**platform-operations 观点**（第379-380行）:
```
退款套现:
- 领券→下单→退款，保留优惠
- 防御措施: 退款后券不返还，或返还后有效期不延长
```
**理由**: 防止薅羊毛和套现漏洞

#### 影响范围
- **影响功能**: 退款流程、券状态管理
- **影响用户**: 所有用券订单的退款用户
- **影响商家**: 所有参与券活动的商家

#### 推荐方案
**分阶段方案（平衡各方利益）**:

```
阶段1: MVP（前3个月）- 简单规则
┌─────────────────────────────────────────────────────┐
│ 未发货退款: 券不返还                                │
│ 已发货退款: 券不返还                                │
│ 理由: 简化逻辑，防止薅羊毛                          │
└─────────────────────────────────────────────────────┘

阶段2: 体验优化（3-6个月后）- 引入场景区分
┌─────────────────────────────────────────────────────┐
│ 未发货退款: 券返还（用户友好）                      │
│ 已发货且未收货: 券返还                               │
│ 已收货/已使用: 券不返还（商家保护）                 │
│ 理由: 平衡用户体验和商家利益                        │
└─────────────────────────────────────────────────────┘

阶段3: 精细化运营（6个月后）- 引入信用机制
┌─────────────────────────────────────────────────────┐
│ 高信用用户（退款率<5%）: 优先返还券                │
│ 异常用户（退款率>20%）: 券不返还                    │
│ 理由: 激励良好用户，防范恶意用户                    │
└─────────────────────────────────────────────────────┘
```

#### 是否需要业务方确认
✅ **是** - 这是核心业务规则，影响用户满意度和商家成本，需要业务方权衡。

### 🔴 冲突2: 成本分摊比例

#### 涉及专家
- **platform-operations**: 平台全额补贴起步（3个月）
- **seller-domain**: 梯度分摊，商家可选0%-100%

#### 各方观点

**platform-operations 观点**（第131-137行）:
```
阶段一：平台全额补贴（MVP期，前3个月）
- 平台承担100%
- 目的：快速验证效果，降低商家抵触
- 适用范围：全场通用券

阶段二：平台+商家灵活分摊（3-12个月）
- 模式A: 100%:0% - 平台大促活动、新人券
- 模式B: 70%:30% - 品类券、商家自主活动
- 模式C: 50%:50% - 商家定制券、高毛利品类
```
**理由**: 
- 降低商家参与门槛，快速启动
- 避免因分摊问题导致商家不参与

**seller-domain 观点**（第52-82行）:
```
推荐方案: 梯度分摊机制(方案C)

核心设计理念: 给商家选择权,让不同类型商家按需参与

商家参与选项:
○ 不参与(默认选项)
○ 参与但平台承担全部成本(需满足平台流量考核要求)
○ 参与-我承担10%(平台流量扶持+0.5倍系数)
○ 参与-我承担30%(平台流量扶持+1倍系数)
○ 参与-我承担50%(平台流量扶持+2倍系数)
○ 参与-我承担100%(平台流量扶持+5倍系数)
```
**理由**:
- 不同商家类型对成本的承受能力差异巨大（品牌商家愿意承担100%，个体商家只能承受0%）
- 强制分摊会导致大量商家拒绝参与
- 通过流量扶持激励商家主动承担成本

#### 影响范围
- **影响功能**: 券配置后台、结算系统、对账系统
- **影响商家**: 所有商家的参与意愿和成本
- **影响平台**: 平台补贴成本预算

#### 推荐方案
**融合方案（分阶段实施）**:

```
MVP阶段（前3个月）:
┌─────────────────────────────────────────────────────┐
│ 默认: 平台100%承担                                  │
│ 可选: 商家可主动选择承担30%/50%/100%               │
│       （选择承担的商家获得流量扶持）                │
│ 理由: 降低门槛，同时探索商家意愿                    │
└─────────────────────────────────────────────────────┘

扩展阶段（3-12个月）:
┌─────────────────────────────────────────────────────┐
│ 引入梯度分摊（0%/10%/30%/50%/100%）                │
│ 流量扶持系数:                                      │
│   - 0%承担 → 1倍流量（基础）                        │
│   - 30%承担 → 2倍流量                               │
│   - 50%承担 → 3倍流量                               │
│   - 100%承担 → 5倍流量                              │
│ 理由: 既有默认保障，又有激励机��                    │
└─────────────────────────────────────────────────────┘

成熟阶段（12个月后）:
┌─────────────────────────────────────────────────────┐
│ 动态分摊: 基于商家等级、历史ROI、品类毛利           │
│       动态推荐最优分摊比例                          │
│ 理由: 精细化运营，效率最大化                        │
└─────────────────────────────────────────────────────┘
```

#### 是否需要业务方确认
✅ **是** - 影响平台成本预算，需要确认平台最高承担比例和流量扶持力度。

### 🔴 冲突3: 券的有效期设置

#### 涉及专家
- **platform-operations**: 分层建议（新人7天、沉睡3天、活跃24小时）
- **buyer-domain**: 未明确建议，但提到需要"过期前1天+1小时双提醒"
- **seller-domain**: 未涉及

#### 各方观点

**platform-operations 观点**（第29-51行）:
```
A. 新用户（注册≤7天）
- 有效期: 7天
- 理由: 制造紧迫感，短期内刺激决策

B. 沉睡用户（30-90天未下单）
- 有效期: 3天
- 理由: 短有效期制造紧迫感

C. 活跃用户（近7天有下单）
- 有效期: 24小时
- 理由: 极短有效期刺激即时消费
```
**理由**: 
- 新人需要时间了解平台，7天合理
- 沉睡用户召回，短有效期促立即行动
- 活跃用户避免产生"等券"心理

**buyer-domain 观点**（第427-433行）:
```
| 提醒时机 | 渠道 | 提醒内容 | 频率限制 |
| 过期前1天 | Push + 站内信 | "您的5元券将于明天过期" | 1次/券 |
| 过期前1小时 | Push | "紧急提醒：您的5元券还有1小时过期" | 1次/券 |
```
**暗示**: 需要至少提前1天提醒，说明3天/24小时可能过短

#### 影响范围
- **影响功能**: 券发放系统、提醒系统
- **影响用户**: 券的使用率、过期率
- **影响指标**: 核销率、ROI

#### 推荐方案
**分层有效期 + 提前提醒**:

```
┌─────────────────────────────────────────────────────┐
│ 用户类型      │ 有效期  │ 提醒时机                  │
├─────────────────────────────────────────────────────┤
│ 新用户        │ 7天     │ 过期前1天 + 过期前1小时  │
│ 沉睡用户      │ 5天     │ 过期前1天 + 过期前1小时  │
│ 活跃用户      │ 3天     │ 过期前1天 + 过期前1小时  │
└─────────────────────────────────────────────────────┘

调整说明:
- 沉睡用户从3天延长至5天（buyer-domain提醒需要）
- 活跃用户从24小时延长至3天（避免过短导致高过期率）
- 所有类型统一"提前1天+1小时"双提醒
```

#### 是否需要业务方确认
⚠️ **建议确认** - 有效期影响核销率和成本，可通过A/B测试验证。

## 三、待确认问题清单

### Q1: 退款时券是否返还？
**影响**: 用户满意度 vs 商家成本

**选项**:
- **A. 返还券**: 未发货时返还，已发货不返还
  - 优点: 用户友好，降低退款顾虑
  - 缺点: 增加商家成本，可能被恶意利用
- **B. 不返还券**: 任何情况都不返还
  - 优点: 保护商家利益，防止薅羊毛
  - 缺点: 用户体验差，可能降低下单转化

**推荐**: **方案A（分场景返还）**
- 理由: 平衡用户和商家利益
- 实施建议: MVP期先不返还，观察数据后优化

---

### Q2: 成本分摊比例？
**影响**: 平台预算、商家参与度

**选项**:
- **A. 平台100%承担**: 
  - 优点: 商家零压力，参与度高
  - 缺点: 平台成本高，不可持续
- **B. 梯度分摊（0%/10%/30%/50%/100%）**: 
  - 优点: 灵活，商家可选
  - 缺点: 系统复杂，对账麻烦
- **C. 固定分摊（平台70%/商家30%）**: 
  - 优点: 简单，对账清晰
  - 缺点: 部分商家不参与

**推荐**: **方案B（梯度分摊）**
- 理由: 兼顾灵活性和参与度
- 实施建议: MVP期默认0%（平台100%），3个月后引入梯度分摊

### Q3: 券的有效期设置？
**影响**: 核销率、用户压力感知

**选项**:
- **A. 统一7天**: 
  - 优点: 简单，用户有足够时间
  - 缺点: 缺乏紧迫感，可能降低转化
- **B. 分层有效期（新人7天/沉睡3天/活跃24小时）**: 
  - 优点: 精细化运营
  - 缺点: 复杂，活跃用户可能觉得过短
- **C. 统一3天**: 
  - 优点: 平衡紧迫感和使用时间
  - 缺点: 新人可能不够时间了解平台

**推荐**: **方案B（但调整为：新人7天/沉睡5天/活跃3天）**
- 理由: 兼顾精细化和用户友好
- 实施建议: 通过A/B测试验证最优有效期

### Q4: 是否支持券叠加？
**影响**: 系统复杂度、成本控制

**选项**:
- **A. 不支持叠加**: 
  - 优点: 简单，成本可控
  - 缺点: 优惠力度有限
- **B. 支持叠加（最多2张）**: 
  - 优点: 提升优惠力度
  - 缺点: 系统复杂，成本增加
- **C. 支持叠加但限制类型（无门槛券不叠加）**: 
  - 优点: 平衡体验和成本
  - 缺点: 规则复杂

**推荐**: **方案A（MVP不支持，第二期考虑方案C）**
- 理由: MVP简单优先，后续迭代优化

### Q5: 平台月预算上限？
**影响**: 成本控制、发放规模

**选项**:
- **A. 50万/月**: 
  - 可支持: 5万张新人券（10元）或 10万张活跃券（5元）
- **B. 100万/月**: 
  - 可支持: 10万张新人券或 20万张活跃券
- **C. 按ROI动态调整**: 
  - 规则: ROI≥3时自动增加预算，ROI<2时自动减少

**推荐**: **方案C（设置50万起步，根据ROI动态调整）**
- 理由: 数据驱动，避免预算浪费
