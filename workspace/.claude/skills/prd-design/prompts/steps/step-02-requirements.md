# 步骤 02：需求收集

## 强制执行规则（首先阅读）

### 通用规则
- 🛑 绝不在没有用户输入的情况下生成内容
- 📖 关键：在采取任何行动之前阅读完整的步骤文件
- 🔄 关键：使用 'C' 加载下一步时，确保读取整个文件
- 📋 你是引导者，不是内容生成器
- ✅ 你必须始终用中文输出

## 执行协议
- 🎯 在采取任何行动之前显示你的分析
- 💾 仅在用户选择 C（继续）时保存
- 📖 更新 frontmatter，将此步骤添加到 stepsCompleted
- 🚫 禁止在 C 被选择之前加载下一步

## 🚨 系统成功/失败指标

### ✅ 成功
- 菜单正确显示且用户输入正确处理
- 在继续之前更新 frontmatter
- 在执行之前读取整个下一步文件

### ❌ 系统失败
- 在用户未选择 'C'（继续）的情况下继续
- 未用步骤完成更新 frontmatter
- 仅读取部分步骤文件

---

## 步骤目标

通过协作对话收集产品需求：
1. 了解产品愿景和目标
2. 识别目标用户和用户痛点
3. 分类业务领域（医疗、金融、电商等）
4. 确定领域复杂度（低/中/高）
5. 将信息填入 PRD 文档

## 执行指令

### 1. 读取输入文档

从 frontmatter 读取 `inputDocuments` 列表。

如果有输入文档：
- 使用 Read 工具读取每个文档
- 总结关键信息
- 向用户展示摘要

### 2. 协作对话 - 产品愿景

与用户进行对话，了解：

**问题 1：产品愿景**
"请描述这个产品的核心愿景是什么？要解决什么问题？"

**问题 2：目标用户**
"谁是这个产品的目标用户？他们有什么特征？"

**问题 3：用户痛点**
"用户当前面临哪些痛点？为什么现有解决方案不够好？"

**问题 4：核心目标**
"这个产品的核心目标是什么？如何衡量成功？"

### 3. 业务领域分类

基于收集的信息，分析业务领域：

向用户展示分析：

```
基于你的描述，我分析这个产品属于以下领域：

业务领域：[电商/医疗/金融/教育/平台/其他]
领域特征：[列出识别的特征]

这个分类正确吗？如果不正确，请告诉我应该是什么领域。
```

等待用户确认或修正。

### 4. 确定领域复杂度

基于以下标准评估复杂度：

**低复杂度：**
- 单一用户角色
- 简单的业务流程
- 无特殊合规要求

**中复杂度：**
- 2-3 个用户角色
- 中等业务流程复杂度
- 一些行业标准要求

**高复杂度：**
- 多个用户角色（3+）
- 复杂的业务流程和规则
- 严格的合规/监管要求
- 多边平台或生态系统

向用户展示评估：

```
领域复杂度评估：[低/中/高]

理由：
- [列出评估依据]

这个评估合理吗？
```

等待用户确认或调整。

### 5. 更新 PRD 文档

将收集的信息填入 PRD 文档的"四、项目背景与目标"部分：

```markdown
## 四、项目背景与目标

### 背景描述

- 需求背景：[用户提供的产品愿景]
- 用户痛点：[用户提供的痛点描述]
- 目标用户：[用户提供的目标用户]

### 目标设定

- 核心目标：[用户提供的核心目标]
- 成功指标：[用户提供的衡量标准]
```

更新 frontmatter：
```yaml
stepsCompleted: ["step-01-init", "step-02-requirements"]
currentStep: "step-03-domain-check"
domain: "[识别的业务领域]"
domainComplexity: "[低/中/高]"
lastModified: "[当前时间]"
```

保存文档。

### 6. 确认和继续

向用户展示：

```
✅ 需求收集完成

业务领域：[领域]
复杂度：[复杂度]

已更新 PRD 文档的"项目背景与目标"部分。

下一步：领域专家检查

[A] 高级引导（详细问答）
[P] 派对模式（快速推进）
[C] 继续到步骤 03
```

等待用户输入，然后：
1. 确认 frontmatter 已更新
2. 读取 `steps/step-03-domain-check.md` 完整文件
3. 执行步骤 03
